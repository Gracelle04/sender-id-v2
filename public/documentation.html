01

Vue d'ensemble du projet

Suivi Sender ID — Afrique Ouest & Centrale est une application web interne développée pour LAfricaMobile. Elle permet aux équipes de suivre l'état des demandes de Sender ID auprès des opérateurs téléphoniques dans les pays d'Afrique subsaharienne où LAfricaMobile opère.

Objectif métier

Un Sender ID (identifiant d'expéditeur SMS) est le nom affiché dans la messagerie du destinataire lorsqu'il reçoit un SMS commercial. L'enregistrement d'un Sender ID nécessite une soumission administrative auprès de chaque opérateur dans chaque pays. Cette plateforme centralise et suit ces demandes en temps réel.

Fonctionnalités principales

📊

Tableau de bord

KPIs temps réel : total des Sender IDs, approuvés, à notifier, en attente par opérateur.

🌍

Suivi par pays

15+ pays couverts. Vue filtrée par pays avec accès direct à la vue Zoho Desk correspondante.

📄

Gestion documentaire

Upload et stockage des documents clients sur Cloudinary. Catégories par type et nom client.

📋

Modèles

Bibliothèque de templates de formulaires téléchargeables, versionnés par pays.

🔔

Notifications clients

Marquage des Sender IDs approuvés comme "notifiés au client" avec horodatage.

👥

Authentification & sécurité (JWT + OTP)

L'authentification repose sur un système à deux facteurs : vérification email/mot de passe puis validation d'un code OTP à 6 chiffres envoyé par email. Un JWT est émis uniquement après validation des deux étapes.

Flux d'authentification

ÉtapeAction utilisateurBackendRésultat
1 Saisie email + mot de passe POST /api/auth/login — vérifie bcrypt hash OTP généré, hashé, stocké en BDD avec TTL 10 min, envoyé par email
2 Saisie code OTP à 6 chiffres (auto-submit) POST /api/auth/verify-otp — compare bcrypt JWT signé (8h), suppression de l'OTP
3 Toutes les requêtes suivantes Middleware authRequired — vérifie header Authorization: Bearer <token> Accès accordé ou 401

Flux d'invitation

Les nouveaux utilisateurs sont invités par l'administrateur. Ils n'ont jamais de mot de passe temporaire — le lien d'invitation leur permet de définir directement leur propre mot de passe.

ÉtapeEndpointDescription
Admin invite POST /api/admin/invite Crée compte inactif + token crypto 64 chars + email avec lien valable 48h
Utilisateur clique GET /activate.html?token=... Page d'activation — récupère email via GET /api/auth/invitation/:token
Définition mot de passe POST /api/auth/activate Hash bcrypt du nouveau mot de passe, compte activé, token supprimé

Sécurité des mots de passe et OTP

// Mot de passe — bcrypt avec 10 rounds de salage
const hash = await bcrypt.hash(password, 10);
const ok   = await bcrypt.compare(plaintext, hash);

// OTP — 6 chiffres, hashé bcrypt rounds=6 (rapidité), TTL MongoDB 10 min
const otp = String(Math.floor(100000 + Math.random() * 900000));
await db.collection('otps').insertOne({
  email: user.email,
  otp: await bcrypt.hash(otp, 6),
  expires_at: new Date(Date.now() + 10 * 60 * 1000)
});

// JWT — 8 heures, payload minimal
const token = jwt.sign(
  { id, email, nom, role },
  JWT_SECRET,
  { expiresIn: '8h' }
);
Sécurité : L'OTP est hashé en base de données (pas stocké en clair). Même un accès direct à MongoDB ne révèle pas les codes en cours. L'expiration TTL est gérée nativement par MongoDB via index expireAfterSeconds: 0.

Middlewares d'autorisation

MiddlewareCondition d'accèsRéponse si refusé
authRequired Token JWT valide dans le header Authorization 401 Non authentifié
adminRequired Token valide + role === 'admin' 403 Accès admin requis
writeRequired Token valide + role !== 'viewer' 403 Accès en lecture seule

Implémentation double couche (Frontend + Backend)

// Backend — middleware writeRequired bloque les viewers
function writeRequired(req, res, next) {
  authRequired(req, res, () => {
    if (req.user.role === 'viewer')
      return res.status(403).json({ error: 'Accès en lecture seule' });
    next();
  });
}

// Frontend — masquage des éléments d'écriture pour les viewers
if (user.role === 'viewer') {
  window.__viewerMode = true;
  document.getElementById('tab-form').style.display = 'none';
  ['btn-add-doc', 'btn-add-modele'].forEach(id => {
    const el = document.getElementById(id);
    if (el) el.style.display = 'none';
  });
}
La restriction backend est la barrière de sécurité réelle. La restriction frontend améliore l'expérience utilisateur en masquant les fonctions inutilisables, mais ne remplace pas le contrôle serveur.

Stockage fichiers — Cloudinary

Cloudinary est utilisé comme CDN et service de stockage pour tous les fichiers uploadés par les utilisateurs. Le SDK officiel Node.js v2 est intégré.

Configuration

cloudinary.config({
  cloud_name: process.env.CLOUDINARY_CLOUD_NAME,
  api_key:    process.env.CLOUDINARY_API_KEY,
  api_secret: process.env.CLOUDINARY_API_SECRET,
  secure:     true   // URLs HTTPS uniquement
});

Structure des dossiers Cloudinary

DossierContenuresource_type
sender-id/documents Documents clients (PDF, Word, images, etc.) raw
sender-id/modeles Templates de formulaires à télécharger raw
Le type raw est utilisé pour tous les fichiers car Cloudinary traite les images différemment (transformations automatiques). En raw, le fichier est stocké tel quel et l'URL directe permet le téléchargement sans modification.

Flux d'upload

// Buffer Multer → Stream → Cloudinary
function uploadToCloudinary(buffer, options) {
  return new Promise((resolve, reject) => {
    const stream = cloudinary.uploader.upload_stream(options, (err, result) => {
      if (err) reject(err); else resolve(result);
    });
    Readable.from(buffer).pipe(stream);
  });
}

La réponse Cloudinary contient result.public_id (identifiant pour suppression) et result.secure_url (URL HTTPS pour téléchargement), stockés en base de données.

Service email — Nodemailer + Gmail SMTP

Nodemailer est configuré avec Gmail SMTP en utilisant un App Password Google (mot de passe d'application). Ce mécanisme permet l'envoi d'emails sans activer "accès moins sécurisé" sur le compte Google.

Configuration du transporter

const transporter = nodemailer.createTransport({
  service: 'gmail',
  auth: {
    user: process.env.SMTP_USER,  // akgc1107@gmail.com
    pass: process.env.SMTP_PASS   // App Password Google 16 caractères
  }
});

Emails envoyés

EmailDéclencheurContenu
OTP de connexion POST /api/auth/login (credentials valides) Code à 6 chiffres, design HTML avec fond bleu LAfricaMobile, valide 10 minutes
Invitation POST /api/admin/invite Lien d'activation personnalisé avec token 64 chars, valide 48 heures. Bouton CTA bleu
Re-invitation POST /api/admin/invite/:id/resend Même template invitation, nouveau token

Mode dégradé (SMTP non configuré)

Si SMTP_USER n'est pas défini dans .env, les emails sont simulés — le contenu est affiché dans les logs serveur au lieu d'être envoyé. Utile pour le développement sans accès SMTP.

Gmail App Password : Nécessite l'activation de la vérification en 2 étapes sur le compte Google. Généré dans Compte Google → Sécurité → Mots de passe d'application. Le mot de passe généré est à 16 caractères et doit être stocké dans SMTP_PASS.

Normalisation des noms de pays

Les noms de pays stockés en base peuvent contenir des accents, apostrophes ou espaces variables. La fonction normalizePays() garantit une correspondance robuste.

function normalizePays(nom) {
  return (nom || '')
    .toLowerCase()
    .normalize('NFD')              // décompose les caractères accentués
    .replace(/[̀-ͯ]/g, '') // supprime les diacritiques
    .replace(/['']/g, ' ')          // normalise les apostrophes
    .replace(/\s+/g, ' ')           // normalise les espaces multiples
    .trim();
  // Exemples : "Sénégal" → "senegal", "Côte d'Ivoire" → "cote divoire"
}

Conteneurisation — Docker

L'application est packagée en image Docker pour faciliter le déploiement sur n'importe quelle infrastructure. L'image est basée sur node:20-alpine pour une taille minimale.

Dockerfile

FROM node:20-alpine     # Image de base légère (~50MB)
WORKDIR /app
COPY package.json .
RUN npm install        # Installe les dépendances dans le layer dédié
COPY . .               # Copie le code source
EXPOSE 3000
CMD ["node", "server.js"]
